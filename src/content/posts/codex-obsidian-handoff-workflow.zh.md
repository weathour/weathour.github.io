我现在把自己的长周期工作流固定成一套很简单的结构：**项目 hub note + 项目四件套 + 今日日志 + 会话留档**。

它的目标不是把系统做复杂，而是实现一件很具体的事：

> 下次开始工作时，我只需要给 Codex 一句短指令，Codex 就能自己把上下文接上。

这套模式特别适合多项目并行、跨天推进、跨会话续接的工作，例如论文写作、综述搭建、研究方向探索、工程项目整理和实验复盘。

## 这套模式的核心理念

这套工作流建立在三个原则上：

### 1. 长期记忆、当天执行、跨会话交接必须分层

很多工作流真正的问题，不是“记得不够多”，而是不同层次的信息混在一起。

- 长期有效的信息，应该放在项目主记忆里
- 只和今天有关的执行过程，应该放在今日日志里
- 只为了下一次会话快速接上的内容，应该放在会话留档里

只要这三层不分开，系统就会越来越重，最后反而降低续接效率。

### 2. 项目必须有一个真正可链接的统一锚点

在 Obsidian 里，真正可双链的对象是**笔记文件**，不是文件夹。

所以每个项目都必须有一个统一入口：

- `项目名.md`

也就是我实际会链接的对象：

- `[[项目名]]`

例如：

- `[[YK-RL]]`
- `[[车辆队列中文综述]]`
- `[[重庆重点研发]]`

项目文件夹只是容器，不是双链目标。这个规则一旦固定下来，daily log、session archive、其他项目笔记之间的链接就都会稳定很多。

### 3. Codex 应该负责读取和压缩上下文，而不是让我重复解释

我不希望每次开始新会话都再写一大段背景说明。

更合理的方式是：

- 我只给出一个短指令
- Codex 自己读取项目长期记忆
- Codex 自己读取最近会话留档
- Codex 自己读取今天的日志
- Codex 再开始执行

这也是这套工作流最重要的使用体验目标。

## 具体结构

### 一、项目 hub note

每个项目有一个统一入口：

- `项目名.md`

它承担四个职责：

- 作为项目唯一的双链入口
- 链接到四件套
- 给出一句话状态
- 展示最近日志和最近会话

我现在会把 hub note 设计成一个真正的导航页，而不是大段堆信息的总档案。

## 二、项目四件套

每个项目都维护四个长期文件：

- `项目名 - Prompt.md`
- `项目名 - Plan.md`
- `项目名 - Implement.md`
- `项目名 - Documentation.md`

它们各自分工明确。

### `Prompt.md`

这里放稳定、不常变化的内容：

- 项目目标
- 关键约束
- 非目标
- done 定义

### `Plan.md`

这里放阶段性执行结构：

- 当前阶段
- 里程碑
- 当前 1–3 个下一步
- 验证方式
- stop rules

### `Implement.md`

这里放 Codex 在这个项目里的操作约定：

- 每次开始要先读什么
- 每次结束要更新什么
- 哪类信息应该写到哪个文件
- 哪些信息不要混放

### `Documentation.md`

这是项目当前状态的主记忆页。

这里集中放：

- 当前状态
- 已知事实
- 关键判断
- 当前风险
- 最近进展
- 精确下一步

如果要快速恢复一个项目的上下文，我会优先让 Codex 先读这个文件。

## 三、今日日志

今日日志只负责当天执行，不再承担长期项目定义。

它通常只保留这些内容：

- 我今天最重要的 3 件事
- 我今天推进的项目
- 我今天做了什么
- 我今天的关键判断 / 卡点 / 反思
- 我明天优先做什么
- 下次需要 Codex 接着做什么

换句话说，今日日志是“今天的执行视图”，不是项目主档案。

## 四、会话留档

我还保留一层轻量的跨会话 handoff：

- `YYYY-MM-DD-会话NN.md`

它只做一件事：让下一个会话快速接上。

每条会话留档通常包含：

- 本次会话完成了什么
- 修改了哪些文件
- 形成了哪些关键判断
- 还有哪些未完成 / 风险点
- 下一次应该从哪里继续

它不是长期项目档案，而是“下一次对话的桥”。

## 命名和链接规则

为了让这套体系稳定工作，我把规则固定成下面这样：

- 项目入口统一为 `项目名.md`
- 项目双链统一写成 `[[项目名]]`
- 四件套统一为 `项目名 - Prompt/Plan/Implement/Documentation.md`
- 项目文件夹只做容器，不做双链目标
- 项目 hub note 尽量直接显示最近日志 / 最近会话的 Dataview 结果

这套规则看起来简单，但它决定了 Obsidian 图谱、反向链接和 Codex 续接效率是否稳定。

## 实际使用方式

### 开始工作时

我只需要说：

- `继续推进 YK-RL`
- `切到车辆队列中文综述`
- `继续昨天那个方向判断`

然后 Codex 自己按顺序读取：

1. 项目的 `Documentation.md`
2. 项目的 `Plan.md`
3. 最近一次相关会话留档
4. 当天日志
5. 必要时再读 `Prompt.md`

这样我就不需要手工重新整理背景。

### 结束工作时

我只需要说：

- `今天先到这里，帮我留档`
- `做个交接总结`
- `把今天状态记下来`

然后 Codex 自己完成：

1. 更新项目 `Documentation.md`
2. 必要时更新 `Plan.md`
3. 更新今日日志
4. 写新的会话留档

如果我只想补一条轻量 handoff，我可以直接说：

- `$agent-session-archiver`

如果我想让 Codex 把项目文档、今日日志和会话留档整套一起收尾，我就直接说：

- `$project-log-manager`

## 为什么这套模式适合长周期工作

我现在的工作往往不是短平快任务，而是跨很多天、很多会话的持续推进。

这套模式适合长周期工作的原因在于：

- 项目有了稳定的双链锚点
- 长期记忆和当天执行被拆开
- 跨会话续接有了独立层
- Codex 可以主动读取上下文，而不是等我手工补背景

本质上，这不是在“多记一点”，而是在提高**恢复上下文的速度和稳定性**。

## 结尾

如果用一句话概括这套工作流，我会这样说：

> 我把项目管理变成了一套 Codex 可以主动读取、主动链接、主动续接的项目记忆结构。

对我来说，这套模式最有价值的地方，不是它看起来更整齐，而是它让我在开始下一次工作时，真正可以只说一句话，就把项目接上。
